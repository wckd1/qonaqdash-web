## Tech Stack

- **Frontend**: Vue 3 + Vite + Pinia + Vue Router + Pico CSS
- **Backend**: Go modular monolith, REST JSON API, JWT auth (HS256)

## Architecture Decisions

- **Project structure**: Feature-based — `src/features/{auth,property,guests,bookings}/` each with own views, stores, composables; shared code in `src/shared/` (api client, layouts, common components)
- **CSS architecture**: `main.css` holds `:root` design tokens and **shared** layout/UI (app shell, list tables, side panels, dashboard toolbar, rooms accordion, dialogs, search bar, form section, status chips, panel, `.form-error`, `.invite-info`, etc.); [docs/design.html](docs/design.html) uses main.css and only adds doc-only layout (swatches, section headings). **Recurring `main.app-main > …` layout** (list **`__viewport`**, dashboard body viewport, search, **direct** child **`form-content__viewport`**) stays in **main.css**. **Single-route** shells (e.g. guest detail **`.guest-detail-body`**) belong in that **view’s `<style scoped>`**; use **`:deep()`** to adjust child components (e.g. nested JSONForm **`.form-content__viewport`**). Prefer explicit **`__viewport` / body** class names over **`:first-child`** for scroll regions. Otherwise keep screen-specific rules scoped; avoid duplicating patterns already in `main.css`. Components should use main.css hooks—avoid custom hex/rgba/rem when a token exists.
- **Token storage**: localStorage for both access and refresh tokens; migrate refresh to httpOnly cookie when backend refresh endpoint is added
- **API client**: Axios with interceptors (auto-attach JWT, 401 redirect, error handling)
- **Feature API modules**: Each feature has its own `api.js` (e.g., `src/features/auth/api.js`) encapsulating HTTP calls and response parsing — stores stay thin and call these modules
- **API integration**: When implementing API integration, propose backend changes explicitly when they would help the frontend: (1) **Response shape** — if a different response model (e.g. structure, fields, pagination) would make the frontend simpler, faster, or avoid extra requests, state the proposed change and what the backend should return; (2) **Missing endpoints** — if the feature needs an endpoint/parameter that does not exist, propose it explicitly (method, path, request/response shape) and explain why it is required. Do not work around suboptimal or missing APIs in silence. **All proposed backend changes must be listed in [docs/backend-change-requests.md](docs/backend-change-requests.md)** (endpoint, params, reason, frontend usage).

## UI/UX Design Decisions

- **Design implementation**: Use `/frontend-design` and `/interface-design` skills when working on UI/UX design and implementation
- **Design system**: All colors, typography, spacing, and depth live in [src/assets/main.css](src/assets/main.css) as CSS custom properties. Reference page: [docs/design.html](docs/design.html). Use tokens from main.css in new components; do not introduce hardcoded hex or ad-hoc values.
- **Navigation**: Collapsible **dark** sidebar — Dashboard, Bookings, Guests, and a **Settings** group (Rooms, guest form, booking form under **`/manage/…`**). Sidebar uses `--sidebar-bg`, `--sidebar-text`, `--sidebar-user-menu-bg` etc. in [main.css](src/assets/main.css); see [docs/design.md](docs/design.md) and design.html. **Manage vs list routes**: Main **Guests** / **Bookings** nav highlights must exclude `/manage/guests` and `/manage/bookings` (e.g. `startsWith('/guests') && !startsWith('/manage/guests')`) so form-editor pages don’t highlight the list items. **Settings group**: Collapsible (closed by default); auto-open only on first load if the URL is already under `/manage/`, or when navigating **into** `/manage/` from outside it—not on every move between manage sub-routes, so a manually collapsed section stays closed. When the sidebar rail is **narrow**, the Settings trigger is hidden and manage sublinks stay visible as icons.
- **Interaction pattern (universal)**: List/Grid -> Side panel (summary + quick actions) -> Full page (detail + edit). Consistent for bookings and guests.
- **Visual style**: Cool gray-blue base (`--canvas`), teal accent (`--brand-primary`) on Pico CSS; corporate but joyful; generous whitespace. Typography: Work Sans (`--font-body`) + Manrope (`--font-display`) for headings (Latin + Cyrillic). Main content area: spacing (--content-area-gap), rounded panel (--content-area-radius). See design system tokens and [docs/design.html](docs/design.html) for full reference.
- **Booking status colors**: `--status-confirmed`, `--status-checked-in`, `--status-checked-out`, `--status-canceled` (see main.css and design reference)
- **Grid style**: Cloudbeds/Mews-style reservation timeline on the Dashboard (`ReservationGrid.vue`). **Preset lengths** 7/14/30 are remembered (localStorage + `/?from=&to=`). Preset windows are **forward-biased**: `from` = **yesterday**, `to` = **today + (n−2)** inclusive, so **today is always the 2nd column** and most columns are upcoming; **custom** From/To and URL query can still show any span. **Dashboard range bar** (`DashboardView.vue`): single row—preset `<select>`, From/To with `min`/`max` (To cannot be before From), **`@change` reloads** the grid (no Apply), **Today** reapplies the current preset length anchored to “now”.

## Conventions

- **Feature planning and implementation**: Before implementing a feature or during feature planning, always refer to [docs/design.md](docs/design.md), [docs/design.html](docs/design.html), and [docs/requirements.md](docs/requirements.md) so implementation aligns with design system and requirements.
- **Backend change requests**: Document all proposed API changes (new endpoints, new query params, response shape changes) in [docs/backend-change-requests.md](docs/backend-change-requests.md).
- **Shared components**: If a component is reused in multiple places and can be moved to a shared component, extract it to `src/shared/` (or the appropriate shared location) and reuse it instead of duplicating. When extracting, keep the component style-light: use existing main.css classes and tokens only; remove the same styles from the feature views that previously owned them so main.css remains the single source of truth.
- `docs/` directory contains design and requirements reference — excluded from builds
- Don't create `.md` files unless explicitly requested
- Don't bypass errors with workarounds — fix root causes
- **API errors**: Backend returns `{ "error": "human-readable message" }`. Use `data?.error` (not `data?.message`) in the API client and in feature views when surfacing API error messages.
- **Design doc sync**: When changing UI or design decisions (e.g. sidebar theme, new tokens), update both [docs/design.md](docs/design.md) (decisions + palette list) and [docs/design.html](docs/design.html) (visual reference + swatches) so the design system stays the single source of truth.

## Design system learnings

- **Intent block** in [src/assets/main.css](src/assets/main.css): who (reception, staff), what (manage bookings/guests), feel (warm, calm, teal accent). When adding or changing tokens, keep intent consistent.
- **Content and forms**: Content sections use full horizontal space (no arbitrary max-width). Form inputs stack vertically; form labels use body font size (`label` rule in main.css). Apply these in new views.
- **When reviewing UI** (e.g. with /interface-design): Ensure fallbacks in components match current token values; avoid template-style constraints (e.g. max-width on cards) that contradict full-width content; run token test (names should evoke the product).

## Frontend / Vue learnings

- **Pinia state in views**: Use `storeToRefs(store)` when destructuring store state in components so the view stays reactive to async store updates (e.g. direct navigation vs. coming from another page that already loaded data).
- **Pinia session cache — property lists**: `usePropertyStore` treats unfiltered `fetchRoomTypes` / `fetchRooms` as idempotent per list via internal hydration flags (not `array.length`, so an empty property still loads correctly). Any fetch with non-empty `q` clears **both** flags, because Rooms search can replace both lists with partial results and the dashboard needs a consistent full pair again.
- **Pinia session cache — form definitions**: Guest/booking blank form payloads from `GET …/form` are cached in `useGuestStore` / `useBookingStore`; use `fetchGuestForm({ force: true })` / `fetchBookingForm({ force: true })` when the form settings screen must reload from the server (Reset). After a successful form-definition PUT from settings, call `replaceGuestFormTemplate` / `replaceBookingFormTemplate` so the cache matches the server without skipping updates. Do **not** apply the same pattern to per-entity `fetchGuest(id)` / `fetchBooking(id)`.
- **Guest/booking list views**: No session-level skip caching for `fetchGuests` / `fetchBookings` — server search and date filters make a safe shared “full list” invariant across routes too brittle.
- **Sidebar active link**: Use CSS class `router-link-exact-active` (not `router-link-active`) for nav link styling when the root path is `/` and all app routes are its children—otherwise the home/dashboard link stays active on every page.
- **Pico margin overrides**: In [main.css](src/assets/main.css) we override Pico’s default bottom margins (buttons, block elements, `details`, form elements `input`/`select`/`textarea`) so the app controls spacing; add new Pico overrides there rather than in components.
- **Pico focus ring**: Pico sets `--pico-outline-width: 0.0625rem` on text inputs; root is `0.125rem`. For custom input-like components (e.g. search bar) use `0.0625rem` in the focus box-shadow so the ring matches standard inputs.
- **Feature vs view naming**: A feature (e.g. `property`) can have multiple views. Name the view after the screen: `RoomsView` for the Rooms screen, not `PropertyView`; reserve `PropertyView` or similar for a future “Hotel settings” or property-level screen. **Settings / admin URLs** use the **`/manage/`** prefix (e.g. `/manage/rooms`, `/manage/guests/form`, `/manage/bookings/form`).
- **404 inside layout**: Use a catch-all child route under the main layout (`path: '/:pathMatch(.*)*'`) so unknown routes still render AppLayout with a NotFound view in the content area.
- **Router order (static before dynamic)**: On each URL branch, register **fixed paths before param routes** (e.g. `guests/new` before `guests/:id`). Otherwise a catch-all param (e.g. `:id`) wins first and treats literal segments as parameter values.
- **Accordions (multiple open)**: Use native `<details>` without a controlled `:open` binding so each accordion opens and closes independently.
- **List-with-search parity**: When adding a new list view that has search, follow the same pattern as the reference feature (e.g. Guests): server-side search with `q`, debounced input, `initialLoading` vs `searching`, spinner in the search bar (no full-page loader). Document any needed API changes in [docs/backend-change-requests.md](docs/backend-change-requests.md). Do not implement client-side filtering when the project standard is server-side search.
- **List view layout**: On list-with-search pages, show the **search bar always** (directly under the page header). Then show error message, loading state, or content below. Do not hide the search bar when there is a load error or empty state—users can retry or refine search.
- **Page layout (floating header)**: App shell is a **flex column** (`div.app-layout`): top bar row, then `div.app-layout-body` as a **flex row** (`aside.sidebar` + `main.app-main`). `<router-view />` renders **directly inside `main`**. Each screen uses **`<header class="page-header">`** (h1 + actions; `flex-shrink: 0`) then content siblings with **`gap: var(--space-md)`** on `main`. **List views**: `<search>` (styled as `main.app-main > search`), then **`<section class="list-content">`**: outer section is **`overflow: hidden`** + **`position: relative`** (panel anchor); wrap table/states in **`<div class="list-content__viewport">`** — that element scrolls (`overflow-y: auto` in `main.css`). **Side panel** must be a **sibling after** `.list-content__viewport` inside the section so it is not inside the scrolling subtree (otherwise absolute panels would scroll with the list). **Forms**: `JsonFormEdit` / `JsonFormView` / `JsonFormBuild` wrap the layout in **`<div class="form-content__viewport">`**. **Dashboard**: **`div.dashboard-body__viewport`** inside **`section.dashboard-body`** for the same scroll vs panel split; booking side panel after the viewport. **Booking detail**: same as **`BookingDetailView`** (errors, then **`v-else-if`** with form + **`section-placeholder`**). **Guest detail**: **`.guest-detail-body`** scrolls; **`div.guest-detail-form`** wraps JSONForm + placeholder, then **`section.previous-bookings`** (flex `order` keeps bookings below the form in the scroll). Nested **`.form-content__viewport`** from JSONForm is neutralized in **`GuestDetailView.vue`** scoped **`:deep(.form-content__viewport)`**. **404**: `main.app-main > .not-found` fills and centers. **`main.app-main`** remains `position: relative` for absolute panels.
- **Avoid redundant wrappers**: Don’t wrap the search bar in an extra div (e.g. list-toolbar-bar). Don’t add a top-level section around form/detail content when the form renderer (e.g. JSONForm groups) already provides section structure. **Exception**: **`list-content__viewport`**, **`dashboard-body__viewport`**, **`form-content__viewport`**, **`guest-detail-body`**, and **`guest-detail-form`** are intentional (scrollport vs panel host, JSONForm scroll root, or guest form + bookings column); do not fold list/dashboard scroll into the outer section alone.
- **Detail view: form + secondary block**: **`BookingDetailView`**: header, errors, then JSONForm (and **`section-placeholder`** when form not ready). **Guest detail**: **`<div class="guest-detail-body">`** contains **`guest-detail-form`** (JSONForm + placeholder) then **`section.previous-bookings`** — bookings stay below the form in the same scroll column.
- **Bookings vs Dashboard**: The **reservation grid** (timeline/Gantt) lives on the **Dashboard** (home). The **Bookings** nav points to a **list view** (like Guests): searchable list, row click → side panel, “Open Full Page” → detail. Use `GET /api/bookings` for the list (with `q` for search); `GET /api/bookings/grid` is for the Dashboard grid.
- **Reservation grid UI**: **Room-type** section rows use design tokens for the band (e.g. `color-mix` of `--control-bg` with `--ink-primary`, not flat dark grays). **Range highlight** after drag-select should stay visible while the **cell context menu** is open (derive highlights from menu range as well as active drag state). **New booking from empty cells**: multi-day selection → check-in **first** day 14:00, check-out **last** day 12:00; **single** day → one night, check-out **next** day 12:00. **Sidebar** Dashboard item uses a **layout-dashboard** icon (widget grid), not a house glyph—keep SVG paths valid.
- **JSONForm edit mode**: Edit flow uses `JsonFormEdit` + `edit/LayoutRenderer`, `edit/ControlRenderer`, `edit/ArrayRenderer`; path helpers in `utils.js` (`scopeToPath`, `getValueByPath`, `setValueByPath`, `getSchemaEntry`). Form pages load schema via GET form endpoints; submit sends camelCase payload (flat for guest, `{ guest, booking }` for booking). Detail edit: Save first, Cancel second (`.btn-secondary`) in `.page-header-actions`; Cancel only on detail views, not create views.
- **Form page height**: On form/detail routes, `<main class="app-main">` also gets `.app-main--fit-content` (`overflow: hidden`, `height: fit-content`, `max-height: 100%`) so short pages do not leave a tall empty main column. **`main.app-main--fit-content > .form-content__viewport`** (booking detail, etc.) uses `flex: 1 1 auto` and `min-height: 0` in `main.css`. **Guest detail** applies the same flex on **`.guest-detail-body`** in **`GuestDetailView.vue`** scoped styles so long content scrolls inside `main`.
- **HorizontalLayout**: `.form-view-layout` has `width: 100%` globally. For horizontal layout, add `.form-view-layout--horizontal > .form-view-layout { width: auto; flex: 1 1 0; min-width: 0 }` so two groups sit side-by-side.
- **`form-view-layout__fields` grid**: That container uses `grid-template-columns: auto 1fr` for label/value pairs. Direct children that should span the full width need `grid-column: 1 / -1` (already applied to `.form-view-layout` and `.form-build-control`). The JSONForm **builder** wraps each node in **`.form-build-shell`**—add **`.form-view-layout__fields > .form-build-shell { grid-column: 1 / -1; min-width: 0 }`** or nested horizontal layouts inside a group stay squeezed into the first column.
- **JSONForm WYSIWYG (`JsonFormBuild`)**: **HorizontalLayout** allows at most **two** direct children; **Add** is full-width when the row is empty, narrow tail on the right when there is one child. **Group titles**: `resolveGroupTitle` returns **no synthetic “Main”** for id `main` without a user `label` (same idea as builder `field_*` ids)—use empty title + “Set title” placeholder in the builder where appropriate.
- **Date-time (RFC 3339)**: Store date-time as RFC 3339 (e.g. `toISOString()`). Edit ControlRenderer converts to/from `datetime-local` (local `yyyy-MM-ddTHH:mm`) for display; on input convert back to RFC 3339.
- **Composables returning refs**: When a composable returns a plain object with refs (e.g. `{ loading: ref(false), results: ref([]) }`), accessing `obj.loading` in templates gives the **Ref object** (always truthy), not its `.value`. Wrap the return in `reactive()` so nested refs auto-unwrap: `return reactive({ loading, results, ... })`. This enables template access like `searchContext.loading` to work as a boolean, not a Ref.
- **i18n (vue-i18n)**: Messages in `src/locales/{en,ru}.json` (nested keys by area: `nav.*`, `dashboard.*`, `jsonForm.*`, etc.). Bootstrap: `src/i18n/index.js`; locale preference: `useSettingsStore` + `src/shared/i18n/resolveLocale.js` (pin → `navigator` → `en`). **`document.documentElement.lang`** follows active locale (see `App.vue`).
- **Feature development — no hardcoded UI copy**: For **new or changed features**, all **user-visible** strings (headings, buttons, placeholders, empty/loading/error messages where the frontend supplies the text, `aria-label` / `title`, etc.) must go through **i18n**: add the key to **`src/locales/en.json`** and **`src/locales/ru.json`**, use **`useI18n()` + `t()`** in `<script setup>` or **`$t`** in templates. Reuse existing keys when the wording matches. **Document titles** follow the existing **`pageTitle.*`** / router i18n pattern; detail views may refine **`document.title`** when async data loads (e.g. guest/booking name). **API error bodies** (`data.error`) stay as the server returns until the backend exposes stable codes or localized fields—do not “fix” that by hardcoding English in the client for server messages.
- **i18n outside SFCs**: For shared helpers (e.g. `humanizeGroupId`, date formatting) that are not components, use **`i18n.global.t(...)`** from `@/i18n` so strings stay in the locale files and follow the same active locale.
- **JSONForm builder type line**: Map JSON Schema type/format tokens to **`jsonForm.build.schemaType.*`**; use **`te(key)`** and fall back to the raw token when unknown. Support **`array<…>`** and **` | `** unions. In script logic that also uses **`useI18n().t`**, do not name inner variables **`t`** (e.g. use `x` in `.filter()`).
- **Locale wording**: UI labels are not always literal translations of English—prefer **product-appropriate** terms per locale (e.g. RU badges for VerticalLayout/HorizontalLayout as «группа», not «вёрстка», when that matches staff vocabulary).
